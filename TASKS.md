# 대결 MVP 작업 분해

## Day 3 목표

큰 웹앱 아이디어를 한 번에 구현하지 않고, 우선순위와 작은 작업 단위로 나눈다.

기준:

- **Must**: 첫 배포에 반드시 필요한 기능
- **Should**: 첫 배포에 있으면 좋지만, Must가 끝난 뒤 진행할 기능
- **Later**: 추후 확장 기능

## Must

첫 배포의 핵심 가치인 “사용자가 선택지를 입력하면 개인 성향을 바탕으로 하나를 추천받는다”를 검증하기 위한 기능이다.

1. 프로젝트 기본 화면 구성
2. 회원가입 및 로그인
3. 사전 성향 질문 작성 및 저장
4. 고민 내용과 선택지 입력
5. 추천 생성
6. 추천 결과 표시
7. 추천 채택 또는 다시 추천 반응 저장
8. 이전 추천 결과와 채택률 확인

## Should

Must 기능이 동작한 뒤 첫 사용 경험을 더 좋게 만들기 위한 기능이다.

1. 추천 전 추가 질문 최대 5개
2. 반응형 UI 정리
3. 입력값 검증과 친절한 오류 메시지
4. 추천 기록 상세 페이지
5. 기본 사용 안내 화면

## Later

첫 배포에서는 제외하고, 실제 사용 반응을 확인한 뒤 추가할 기능이다.

1. 상품 가격 자동 비교
2. 식당 자동 검색
3. 여행지 정보 자동 수집
4. 외부 서비스 연동
5. 분야별 전용 추천 엔진
6. 네이티브 모바일 앱
7. 소셜 로그인
8. 비밀번호 찾기

## 운영 준비 문서

구현 작업과 별개로 배포 전에 다음 문서를 유지한다.

- `DEPLOYMENT_CHECKLIST.md` — 환경 변수, 비밀 정보, 배포 전후 확인, 롤백 기준
- `.env.example` — 필요한 환경 변수 이름 예시
- `README.md` — 로컬 실행과 검증 방법

## Must 작업 세부 분해

각 작업은 30~90분 안에 구현하고 확인할 수 있는 단위로 나눈다.

### M1. 프로젝트 첫 화면 만들기

목표:

- 사용자가 앱의 목적을 이해할 수 있는 첫 화면을 만든다.

완료 기준:

- 앱 이름 `대신 결정해드립니다` 또는 `대결`이 보인다.
- 앱이 무엇을 해주는지 한 문장으로 설명한다.
- 로그인 또는 시작 버튼이 보인다.

검증:

- 브라우저에서 첫 화면을 열었을 때 앱의 목적을 바로 이해할 수 있다.
- `npm run test:m1`이 통과한다.
- `npm run build`가 통과한다.

### M2. 사용자 계정 모델 정하기

목표:

- 회원가입과 로그인을 위해 필요한 최소 사용자 정보를 정한다.

완료 기준:

- 이메일을 로그인 식별자로 사용한다.
- 비밀번호를 저장할 수 있는 방식이 정해진다.
- 사용자별 추천 기록을 분리할 수 있다.

검증:

- 사용자 A와 사용자 B의 데이터가 섞이지 않는 구조로 설명할 수 있다.
- `DATA_MODEL.md`에 `auth.users`, `profiles`, `decisions`, `options`, `recommendations`의 역할이 설명되어 있다.
- `DATA_MODEL.md`에 `user_id = auth.uid()` 기준의 데이터 분리 규칙이 설명되어 있다.

### M3. 회원가입 구현

목표:

- 사용자가 이메일과 비밀번호로 계정을 만들 수 있다.

완료 기준:

- 이메일과 비밀번호 입력 폼이 있다.
- 이미 가입된 이메일은 중복 가입되지 않는다.
- 가입 성공 후 로그인 상태가 되거나 로그인 화면으로 이동한다.

검증:

- 새 이메일로 가입할 수 있다.
- 같은 이메일로 다시 가입하면 오류가 표시된다.
- 회원가입 직후 `profiles` row가 자동 생성된다.
- 첫 MVP에서는 이메일/비밀번호 로그인만 사용한다.

### M4. 로그인과 로그아웃 구현

목표:

- 가입한 사용자가 로그인하고 로그아웃할 수 있다.

완료 기준:

- 이메일과 비밀번호로 로그인할 수 있다.
- 로그인하지 않은 사용자는 개인 기록 화면에 접근할 수 없다.
- 로그아웃할 수 있다.

검증:

- 올바른 계정으로 로그인된다.
- 잘못된 비밀번호는 거부된다.
- 로그아웃 후 보호된 화면 접근이 막힌다.

### M5. 사전 성향 질문 만들기

목표:

- 사용자의 선택 성향을 파악하기 위한 기본 질문을 만든다.

완료 기준:

- 최소 5개의 성향 질문이 있다.
- 질문은 구매, 음식, 여행지 등 여러 선택 상황에 공통으로 쓸 수 있다.
- 답변은 사용자별로 저장된다.

검증:

- 로그인한 사용자가 성향 질문에 답할 수 있다.
- 다시 접속해도 답변이 유지된다.

### M6. 고민과 선택지 입력 화면 만들기

목표:

- 사용자가 현재 고민과 두 개 이상의 선택지를 입력할 수 있다.

완료 기준:

- 고민 제목 또는 질문을 입력할 수 있다.
- 선택지를 최소 2개 입력할 수 있다.
- 선택지가 2개 미만이면 추천 요청이 막힌다.

검증:

- `점심 뭐 먹지?` 같은 고민과 2개 이상의 선택지를 입력할 수 있다.
- 선택지가 1개이면 오류가 표시된다.

### M7. 기본 추천 로직 만들기

목표:

- 저장된 성향과 현재 선택지를 바탕으로 하나를 추천한다.

완료 기준:

- 추천 선택지 하나가 생성된다.
- 짧은 추천 이유가 생성된다.
- 나머지 선택지를 고르지 않은 이유가 생성된다.
- 추천 확신도 백분율이 표시된다.

검증:

- 같은 입력에 대해 추천 결과 형식이 항상 유지된다.
- 추천 결과에 빈 항목이 없다.

### M8. 추천 결과 화면 만들기

목표:

- 사용자가 추천 결과를 쉽게 이해할 수 있게 표시한다.

완료 기준:

- 추천한 선택지 하나가 강조된다.
- 추천 이유가 짧게 보인다.
- 제외 이유가 선택지별로 보인다.
- 추천 확신도가 보인다.
- `이 추천이 마음에 들어요`, `다시 추천해주세요` 버튼이 있다.

검증:

- 사용자가 다시 비교하지 않아도 추천 결론을 바로 알 수 있다.

### M9. 추천 반응 저장하기

목표:

- 사용자의 추천 채택 여부를 저장한다.

완료 기준:

- `이 추천이 마음에 들어요`를 누르면 채택으로 저장된다.
- `다시 추천해주세요`를 누르면 미채택으로 저장된다.
- 다시 추천 요청 시 새로운 추천을 받을 수 있다.

검증:

- 추천별 사용자 반응이 저장된다.
- 저장된 반응으로 채택률을 계산할 수 있다.

### M10. 추천 기록과 채택률 화면 만들기

목표:

- 사용자가 과거 추천 결과와 채택률을 확인할 수 있다.

완료 기준:

- 이전 고민 목록을 볼 수 있다.
- 각 고민의 추천 결과를 볼 수 있다.
- 전체 추천 채택률이 백분율로 표시된다.

검증:

- 추천을 여러 번 실행한 뒤 기록 개수와 채택률이 맞게 표시된다.

## 첫 번째 구현 작업 후보

첫 코딩 작업은 **M1. 프로젝트 첫 화면 만들기**가 적합하다.

이유:

- 데이터베이스나 로그인 없이 시작할 수 있다.
- 앱의 목적과 톤을 먼저 확인할 수 있다.
- 이후 화면 설계의 기준이 된다.

첫 작업 요청 예시:

```text
TASKS.md의 M1만 구현하자.
아직 로그인이나 데이터 저장은 만들지 마.
첫 화면에는 앱 이름, 한 문장 설명, 시작 버튼만 넣어줘.
구현 전에 어떤 파일을 바꿀지 먼저 설명해줘.
```
