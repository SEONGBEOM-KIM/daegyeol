# 대결 AI 개발 하네스

## 목적

이 문서는 `대신 결정해드립니다(대결)` 프로젝트를 AI와 함께 반복적으로 개발하기 위한 실행 순서다.

좋은 하네스는 프롬프트 하나가 아니라 다음을 연결한다.

```text
목표 문서
→ 작업 분해
→ 작업 규칙
→ 작은 구현
→ 검증
→ 기록
→ 다음 작업
```

## 기준 문서 역할

| 문서 | 역할 |
|---|---|
| `AGENTS.md` | AI가 따라야 할 작업 규칙 |
| `PROJECT_BRIEF.md` | 프로젝트 목적, 사용자, MVP 범위 |
| `TECH_DECISION.md` | 기술 선택과 tradeoff |
| `TASKS.md` | Must / Should / Later 작업 분해 |
| `M1_TEST_CHECKLIST.md` | 첫 화면 검증 절차 |
| `DAY7_DEBUG_NOTES.md` | 오류 분석 예시 |
| `DEPLOYMENT_CHECKLIST.md` | 배포 전후 확인 절차 |
| `.env.example` | 환경 변수 이름 예시 |
| `README.md` | 실행과 검증 안내 |

## 새 기능 작업 순서

새 기능을 요청할 때는 아래 순서를 따른다.

1. `PROJECT_BRIEF.md`와 맞는 기능인지 확인한다.
2. `TASKS.md`에서 해당 작업이 Must / Should / Later 중 어디인지 확인한다.
3. 구현 전에 바꿀 파일과 이유를 설명한다.
4. 30~90분 안에 끝낼 수 있는 작은 단위로만 구현한다.
5. 기능 완료 기준을 확인한다.
6. 자동 테스트 또는 수동 체크리스트를 추가한다.
7. `npm run build` 등 가능한 검증을 실행한다.
8. 변경 파일, 테스트 결과, 다음 작업을 기록한다.

## 매번 사용할 요청 템플릿

```text
목표:
현재 상황:
이번에 처리할 작은 작업:
제외 범위:
완료 기준:

PROJECT_BRIEF.md, TASKS.md, AGENTS.md를 기준으로 진행해줘.
코드를 작성하기 전에 어떤 파일을 바꿀지 먼저 설명해줘.
작은 변경은 자동으로 진행하고, 중요한 선택은 대안과 장단점을 설명한 뒤 확인받아줘.
작업 후에는 테스트 방법과 실제 검증 결과를 알려줘.
```

## AI가 자동으로 진행해도 되는 일

- 문서 정리
- 오타 수정
- 작은 UI 문구 수정
- 기존 계획에 맞는 작은 코드 변경
- 테스트와 빌드 실행
- 읽기 전용 파일 확인

## 반드시 확인받아야 하는 일

- 기술 스택 변경
- 데이터베이스 구조의 큰 변경
- 인증 방식 변경
- 외부 API 또는 유료 서비스 도입
- 배포 환경 선택
- 첫 배포 범위를 넘는 기능 추가
- 비밀 키, 실제 사용자 데이터, 배포 설정 변경

## 오류가 났을 때 절차

오류가 나면 바로 큰 수정을 하지 않고 아래 순서로 기록한다.

```text
재현 단계
→ 예상 결과
→ 실제 결과
→ 오류 메시지
→ 가장 가능성 높은 원인
→ 최소 수정
→ 재검증
```

## 현재 가능한 검증 명령

```bash
npm run test:m1
npm run build
```

개발 서버:

```bash
npm run dev
```

브라우저:

```text
http://localhost:3000
```
