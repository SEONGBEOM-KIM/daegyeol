# 기술 선택 결정: Next.js + Supabase

## 결정

첫 MVP는 다음 기술 스택으로 만든다.

```text
Next.js
Supabase Auth
Supabase PostgreSQL
OpenAI Responses API
Vercel 또는 Supabase 호환 배포 환경
```

## 선택 이유

사용자는 Supabase에 관심이 있고, 이번 프로젝트를 통해 AI뿐 아니라 현대적인 웹앱 개발 흐름도 함께 배우고 싶다.

`대결` MVP에는 다음 기능이 필요하다.

- 회원가입 및 로그인
- 사용자별 성향 정보 저장
- 추천 요청과 결과 저장
- 추천 채택률 계산
- 추후 반응형 웹앱과 모바일 앱 확장 가능성

Supabase는 인증과 PostgreSQL 데이터베이스를 함께 제공하므로 이 요구사항과 잘 맞는다. Next.js는 반응형 웹앱을 만들고, 나중에 서버 기능과 프론트엔드 구조를 확장하기 좋다.

## 비교한 대안

### 1. Django 중심 구성

장점:

- 사용자가 이미 Python과 Django 경험이 있다.
- 로그인과 서버 렌더링을 빠르게 구현할 수 있다.
- 학습 난이도가 가장 낮다.

선택하지 않은 이유:

- 사용자가 Supabase에 관심이 있다.
- 이번 프로젝트에서 새로운 현대 웹앱 스택을 경험하는 학습 가치가 크다.
- 추후 모바일 확장과 프론트엔드 중심 UI 학습에는 Next.js가 더 적합하다.

### 2. Django API + React/Next.js 프론트엔드

장점:

- 백엔드와 프론트엔드를 분리하는 실무 구조를 배울 수 있다.
- 장기 확장성이 높다.

선택하지 않은 이유:

- 첫 MVP에는 과하게 복잡하다.
- 인증, API, CORS, 프론트엔드 상태관리, 배포를 동시에 배워야 한다.
- 작은 단위로 완성하며 배우는 현재 학습 목표와 덜 맞는다.

## Tradeoff

### 얻는 것

- Supabase Auth를 통한 인증 학습
- PostgreSQL 기반 데이터 모델링 경험
- Next.js 기반 반응형 UI 개발 경험
- 추후 모바일 앱 또는 API 확장에 유리한 구조
- AI 추천 결과를 프론트엔드에서 자연스럽게 표시하는 경험

### 감수하는 것

- Django보다 초기 학습량이 많다.
- JavaScript 또는 TypeScript, React, Next.js, Supabase 개념을 함께 배워야 한다.
- 인증과 데이터 접근 규칙을 올바르게 설정해야 한다.
- OpenAI API 키를 브라우저에 노출하지 않도록 서버 측 호출 구조를 지켜야 한다.

## 첫 MVP에서의 기술 사용 범위

첫 버전에서는 기술을 넓게 쓰지 않고 핵심 기능에 필요한 만큼만 사용한다.

### Next.js

- 화면 구성
- 페이지 이동
- 입력 폼
- 추천 결과 표시
- 반응형 UI

### Supabase

- 이메일 기반 회원가입 및 로그인
- 사용자별 성향 답변 저장
- 추천 요청 저장
- 추천 결과 저장
- 추천 채택 여부 저장

### OpenAI Responses API

- 사용자의 성향, 고민, 선택지를 바탕으로 추천 결과 생성
- 추천 결과는 구조화된 형식으로 받는다.
- 결과에는 추천 선택지, 추천 이유, 제외 이유, 확신도가 포함된다.

## 보안 원칙

- OpenAI API 키는 브라우저 코드에 넣지 않는다.
- Supabase 접근 권한은 사용자별 데이터가 분리되도록 설정한다.
- 중요한 인증 방식 변경이나 외부 API 도입은 먼저 확인받는다.
- 첫 배포에서는 외부 상품, 식당, 여행지 데이터 연동을 하지 않는다.

## Supabase 초기 결정

- Supabase 프로젝트를 실제로 만든다.
- 첫 MVP에서는 이메일/비밀번호 로그인만 사용한다.
- 회원가입 직후 `profiles` row를 자동 생성한다.
- 소셜 로그인과 비밀번호 찾기는 첫 배포에서 제외한다.

- 선택하지 않은 대안과 이유를 기록했다.
