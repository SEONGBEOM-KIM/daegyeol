# 대결 프로젝트 작업 규칙

이 문서는 `대신 결정해드립니다(대결)` 프로젝트에서 AI 에이전트가 따라야 할 작업 규칙이다.

## 프로젝트 목표

선택에 어려움을 느끼는 사용자가 여러 선택지 중 하나를 고를 수 있도록, 개인 성향과 현재 입력을 바탕으로 하나의 선택지를 이유와 함께 추천하는 반응형 웹앱을 만든다.

## 기준 문서

작업 전 다음 문서를 우선 확인한다.

1. `PROJECT_BRIEF.md` — 프로젝트 목적, 사용자, MVP 범위, 성공 기준
2. `TASKS.md` — Must / Should / Later 작업 분해와 구현 순서
3. `AI_10_DAY_LEARNING_PLAN.md` — 학습 진행 방식

## 작업 원칙

- 학습을 우선한다.
- 큰 기능을 한 번에 만들지 않고 작은 단위로 구현한다.
- 첫 배포는 `TASKS.md`의 Must 기능을 우선한다.
- 새로운 기능을 추가하기보다 기존 계획의 핵심 흐름을 먼저 완성한다.
- 어려운 개념은 초보자도 이해할 수 있게 설명한다.
- 코드를 작성하기 전에 항상 어떤 파일을 바꿀지 먼저 설명한다.
- 기능 하나가 끝날 때마다 테스트 방법을 알려준다.
- 작업 후에는 실제로 실행한 검증 결과 또는 실행하지 못한 이유를 알려준다.

## 자동으로 진행해도 되는 작업

다음 작업은 작고 되돌릴 수 있는 범위라면 별도 확인 없이 진행해도 된다.

- 문서 정리
- 오타 수정
- 작은 UI 문구 수정
- 기존 계획에 맞는 작은 코드 변경
- 테스트 또는 린트 실행
- 작업 결과 확인을 위한 읽기 전용 파일 조회

## 확인받아야 하는 결정

다음 작업은 진행 전에 대안과 장단점을 설명하고 확인받는다.

- 기술 스택 선택 또는 변경
- 데이터베이스 구조의 큰 변경
- 인증 방식 변경
- 외부 API 또는 유료 서비스 도입
- 배포 환경 선택
- 첫 배포 범위를 넘어서는 기능 추가
- 삭제, 초기화, 마이그레이션 등 되돌리기 어려운 작업

## 첫 배포에서 제외할 기능

다음 기능은 첫 배포에서 제외한다.

- 상품 가격 자동 비교
- 식당 자동 검색
- 여행지 정보 자동 수집
- 외부 서비스 연동
- 분야별 전용 추천 엔진
- 네이티브 모바일 앱
- 소셜 로그인
- 비밀번호 찾기

## 검증 규칙

각 기능이 끝나면 다음을 확인한다.

1. 완료 기준을 충족했는가?
2. 사용자가 직접 확인할 수 있는 테스트 방법이 있는가?
3. 자동 테스트, 린트, 타입체크, 빌드 중 실행 가능한 검증을 수행했는가?
4. 검증하지 못한 항목이 있다면 이유를 명확히 남겼는가?

최종 보고에는 다음을 포함한다.

- 변경한 파일
- 구현한 내용
- 실행한 테스트와 결과
- 남은 위험 또는 다음 작업
